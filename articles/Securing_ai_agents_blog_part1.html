Your Sign-In Logs Already Know It's an Agent. The Gaps Are in the Layers Above.

AI agents don't break your authentication. They break your security team's ability to reconstruct what happened and why. Here's where the visibility ends, exactly.
April 2026  ·  12 min read  ·  For security engineers, security architects, AppSec, GRC, and auditors
Prefer YouTube? Watch on YouTube ↗

Here's a question that sounds simple but isn't:

If your AI copilot accesses sensitive data, makes a judgment call, and takes an action on behalf of a user — can your security team reconstruct what happened and why?

If you work in IAM, you want to know whose credentials were used. If you're in AppSec, you want to know which tools were invoked and whether input validation held. If you're in SecOps, you want the correlated event chain. If you're in GRC, you want auditable evidence of accountability. If you're an auditor, you want all of the above.

The surprising part? Your identity infrastructure actually answers its piece of this correctly today. Entra ID's sign-in logs can distinguish a Copilot-initiated action from a user-initiated one. The authentication layer works.

The problem is what happens above the identity layer — three critical gaps in your security observability that didn't matter when automation was deterministic, but matter enormously when AI agents are making judgment calls.

To see those gaps clearly, we need to understand the cast of characters involved every time an agent acts on behalf of a user.

Five entities, not two

Most people think of agent actions as a simple pair: the user and the agent. In reality, there are five distinct entities involved in every agent-mediated action. Understanding them is the prerequisite to understanding the accountability gaps.

Visual 1 — The Cast of Characters
E1
The Human
The user with an identity in your IdP. Has roles, permissions, and an audit trail.
E2
App Registration
An enterprise app in your Entra ID tenant. This is what your security infrastructure authenticates — not the human, not the LLM.
E3
OAuth Token
"App X on behalf of User Y with scopes Z." The credential that hits the API.
E4
Decision Engine
The logic that decides what to do. A workflow engine or an LLM. This is where the two paths diverge.
E5
Execution Layer
The runtime that takes decisions and makes actual API calls using the OAuth token.
Entities E1, E2, E3, and E5 are architecturally identical whether you're using Power Automate or an AI copilot. Entity E4 — the decision engine — is the only thing that changes. That's the entire story.

But that one change — E4 — transforms the entire security picture. To see why, we need to double-click on how E4 and E5 relate to each other in each architecture.

The agent framework sees everything — momentarily. It sees the LLM's tool-call requests, executes them, and feeds results back. This is the orchestration layer. It's where security controls could be enforced: tool-level auth, action logging, decision capture. Today, most frameworks don't expose these signals to your security stack.
On the left, decision and execution are one entity — inspectable, static, auditable. On the right, they're split: the LLM decides, the agent framework executes. The framework is the only place where the full decision chain exists. When we say "orchestration layer" throughout this post, we mean E5 — the agent framework that sits between the LLM and your APIs.

Notice what's NOT visible to your security infrastructure in the right-side architecture: the LLM itself (E4) has no identity. It has no Entra ID, no credentials, no OAuth registration. It's a compute function called by the agent framework. Your SIEM has never seen it. Your IAM has never authenticated it. Your DLP has never classified its outputs. It operates entirely inside E4.

This matters because it means the most consequential actor in the chain — the one making decisions — is invisible to your entire security stack. Let's see what this looks like in practice.

Same action, two architectures

Let's take a concrete scenario. Sarah is an IT service desk manager. She wants to automate ticket routing: when a new ticket comes into ServiceNow, route it to the right team and notify the on-call channel if it's high priority.

She can do this two ways: build a Power Automate flow, or ask Copilot to handle it. Let's trace the same action through both paths and watch exactly where the audit trail diverges.

Visual 2 — The Side-by-Side Comparison
Step ⚡ Power Automate 🤖 AI Copilot
1. Sarah initiates Builds a flow in the designer: "If category = network → assign to Network Team. If priority = P1 → post to on-call channel." LOGGED Says: "Monitor new tickets and route them to the right team based on the content." LOGGED
2. Decision is made Flow checks the category field. It's "network." Deterministic match. The outcome was defined at design time by Sarah. INSPECTABLE LLM reads the ticket: "VPN drops on 3rd floor." Is this network? Endpoint? Facilities? The model makes a judgment call. NOT LOGGED
3. Authentication Entra ID sign-in log: non-interactive sign-in, client app = "Power Automate", user = sarah@acme.com LOGGED Entra ID sign-in log: non-interactive sign-in, client app = "Microsoft 365 Copilot", user = sarah@acme.com LOGGED
4. Token scope Scoped to exactly what the flow needs: ServiceNow.ReadWrite, Teams.Post SCOPED Broad scope — might need any of: ServiceNow.ReadWrite, Teams.Post, Mail.Read, Files.Read, Calendar.Read... OVER-SCOPED
5. Execution One API call to assign ticket. One API call to post notification. Two events, one flow. CORRELATED Five API calls: read ticket, search knowledge base, check on-call schedule, look up requester's VIP status, assign ticket. Five separate audit events with no correlation. FRAGMENTED
Look at Step 3 — authentication. It's green on both sides. Your identity infrastructure correctly identifies the acting application in both cases. Now look at Steps 2 and 5. That's where your security telemetry goes dark — and not just for IAM. Your SIEM, DLP, and audit trail all hit the same wall.

Step 3 is the key insight. Your identity infrastructure already distinguishes Power Automate from Copilot from Sarah herself. The "client application" field in sign-in logs tells you which entity authenticated. This isn't new, and it isn't broken.

So if authentication works, what doesn't?

Three gaps in your security observability

The audit trail breaks at exactly three points — none of which are in the identity layer. All three exist in Power Automate too, in theory. But Power Automate's deterministic architecture makes them non-issues. An LLM's non-deterministic architecture makes them critical. Each gap affects different security functions.

Gap 1 — Decision Traceability  ·  Affects: AppSec, SecOps, GRC

The question: WHY did the agent take this action?

With Power Automate, you open the flow definition: category = "network," so it routed to Network Team. Deterministic. Inspectable. Reproducible. With an AI agent, the LLM's reasoning existed only during that specific inference call. Why did it classify "VPN drops on 3rd floor" as a network issue and not a facilities issue? That judgment is not logged anywhere the customer can access. There is no static artifact to inspect. AppSec teams can't validate decision logic. SOC analysts can't reconstruct incident chains. Auditors have no evidence trail.

Gap 2 — Intent Classification  ·  Affects: IAM, Data Security, GRC

The question: did the USER ask for this, or did the AGENT decide on its own?

With Power Automate, every action is user-directed. The flow only does what Sarah designed. There is no ambiguity. With an AI agent, the model may autonomously decide to access additional resources to "better" answer the request. Sarah said "route the ticket." The model decided on its own to also check the knowledge base, look up VIP status, and review the on-call schedule. Were those accesses user-directed or model-initiated? The audit log shows "Copilot on behalf of Sarah accessed knowledge base." It doesn't say whether Sarah asked for that or the model decided it needed it. IAM can't scope permissions by intent. DLP can't distinguish authorized access from autonomous exploration. GRC can't assign accountability.

Gap 3 — Action Chain Correlation  ·  Affects: SecOps, Data Security, Audit

The question: WHICH accesses belong to the same decision sequence?

With Power Automate, one flow execution produces a correlated set of actions. You can trace them as a single unit. With an AI agent, five separate API calls appear as five unrelated audit events: "Copilot accessed ServiceNow," "Copilot accessed knowledge base," "Copilot accessed Teams." A SOC analyst investigating an incident sees five disconnected log entries. Nothing ties them together as one reasoning chain. They cannot reconstruct the decision sequence. SIEM correlation rules can't group them. Data security tools can't evaluate the aggregate sensitivity of the combined access.

What's NOT a gap

Identity-level distinguishability. Your sign-in logs already separate agent actions from user actions via the client application field. If your SIEM rules don't parse this field today, start there — it's low-hanging fruit. But don't confuse having this signal with having full security observability. Knowing that an agent acted is not the same as knowing why it acted, whether the user intended it, or which accesses were part of the same sequence. Every security function — IAM, SecOps, DLP, AppSec, GRC — needs answers to those questions, and none of them can get answers from the identity layer alone.

Where the answers live — and why they can't reach your security stack

If the sign-in log isn't the right place to capture decision traceability, intent classification, and action chain correlation — where should they come from?

The orchestration layer. This is the code that sits between the LLM and the APIs (Entity E5 in our model). At this layer, the full picture exists momentarily: Sarah's original prompt, the model's reasoning about which tools to call, the sequence of API calls, the data returned, and the final action taken.

The orchestration layer could log all of this. The question is: does it? And even if it does, can your security stack — your SIEM, your IAM policy engine, your DLP, your audit system — consume those signals?

Today, the answer to both is mostly no.

The identity and access layer (Layer 3) works. It correctly identifies agents and enforces scopes. But every security tool that depends on it — SIEM, DLP, audit systems, GRC platforms — inherits the same limitation: they don't know why the agent is acting, whether the user intended it, or how this access relates to other accesses in the same reasoning chain. Those answers exist in Layer 2, but no standard interface connects them.

This is the core architectural gap. Your entire security stack is a consumer of decision context. The orchestration layer is the producer. But no standard interface connects them.

Traditional automation never needed this interface because the decision logic was static and inspectable outside the execution path. You could audit a Power Automate flow by reading its definition — you didn't need the execution layer to explain itself at runtime.

AI agents need this interface because the decision logic is dynamic, ephemeral, and exists only at runtime. If the orchestration layer doesn't capture and transmit it, it's gone — and every security function is flying blind.

The Punchline

The agent security problem isn't that your identity infrastructure can't tell agents from users — it can.

The problem is that traditional automation delegated execution of pre-defined decisions, while AI agents delegate the decision-making itself.

Same OAuth plumbing. Same audit log format. Same sign-in log entries. Fundamentally different security observability model.

The fix requires connecting the orchestration layer's decision context to your security stack — IAM for authorization, SIEM for detection and correlation, DLP for sensitivity evaluation, and GRC for audit evidence. That interface doesn't exist today in any standardized form.

What this means for you

Security Engineer

Start parsing the client application field in Entra ID sign-in logs and build SIEM correlation rules that distinguish agent-initiated access from user-initiated access. That's your low-hanging fruit. Then push your agent platform vendors for action chain correlation IDs — a session or trace ID that ties multiple API calls to a single agent reasoning sequence.

Security Architect

The delegation model needs a new dimension: actor type. The same OAuth scope should trigger different policy evaluation depending on whether a deterministic workflow or a non-deterministic model is using it. Design for orchestration-layer signal consumption — your IAM, SIEM, and DLP all need decision context from the agent runtime, not just access events.

AppSec Engineer

Treat agent tool definitions as an attack surface equivalent to API endpoints. Each tool binding is a capability the model can invoke — validate that tool-level input sanitization, output filtering, and authorization checks exist in the orchestration layer, not just in the downstream service. Prompt injection via tool inputs is your top threat vector.

GRC Professional

Your control framework needs a new control category: decision traceability for autonomous systems. This sits between your existing access control and audit logging controls. It asks: can the organization explain why an automated action was taken, not just that it was authorized? Map this to your existing framework — NIST CSF, ISO 27001, or SOC 2 — as a gap analysis.

Auditor

Your evidence collection must expand beyond identity and access logs. Request orchestration layer logs — the reasoning chain, tool invocations, and data accesses that led to each agent action. If the organization can't produce them, that's a finding: insufficient audit trail for autonomous agent actions. Define what "sufficient evidence" means for AI agent decisions in your audit criteria.

The answer — authentication works, but decision traceability, intent classification, and action chain correlation do not — is the starting point.
