Not All AI Agents Are Built the Same. Your Security Controls Assume They Are.

An AI agent is the LLM plus the framework around it — the orchestration code that controls what the model sees, what it can do, and what gets logged. That framework comes in four distinct forms. Each one changes your security posture completely.

Part 1 → Your Sign-In Logs Already Know It's an Agent — The Gaps Are in the Layers Above

Have you ever wondered why Perplexity gives you a different answer than Claude.ai, even when both are using the same underlying Claude model? Or why Microsoft Copilot — which also calls GPT and Claude — feels nothing like either of them? Same models. Completely different behaviour. The difference is what's between you and the model.

That layer is the agent framework, and Part 1 of this series established that the three gaps that matter — decision traceability, intent classification, action chain correlation — all live there. Your identity layer correctly identifies agent traffic. It just can't see why the agent did what it did. This article maps the framework layer. Not to tell you which framework is best, but to show you that there isn't one of them — there are four distinct types running in your enterprise right now, each with different security properties — and what you can actually enforce at design time, at runtime, and after the fact depends entirely on which type you're dealing with.

What "agent" actually means — and what the framework controls

An AI agent is the LLM plus the framework around it. The framework is the orchestration code wrapped around the LLM. When HuggingFace open-sources smolagents, when Microsoft ships their Agent Framework, when Anthropic publishes the Claude Agent SDK — these toolkits are the framework, not the agent. Concretely: when you instantiate smolagents' CodeAgent with a model and a set of tools, that running instance — the LLM together with the orchestration code wrapping it — is the agent. The model is the brain; the framework is the nervous system; the agent is the organism. This article uses agent for the complete system and agent framework for the orchestration toolkit alone — consistent with how Anthropic, HuggingFace, and OpenAI all define the terms. In Part 1's terms, this entire layer is E5, and it's where every architectural choice that affects your security posture is made.

Here is what the framework concretely controls — and why each of these is a security surface, not just an engineering concern.

Visual 1 — What the Agent Framework Controls
Context
What the model sees
System prompt, conversation history, and retrieved documents assembled by the framework pipeline. Some steps are deterministic — applying a prompt template, fetching history. Others involve their own model calls — embedding queries, retrieving and reranking document chunks. The primary model reasons on whatever the pipeline assembles. It has no visibility into what was excluded or how retrieval decisions were made.
Tools
What the model can do
Tool definitions, permissions, and execution scope. The framework presents available tools to the model, validates tool call requests before they execute, and handles results. The model decides which tool to call. The framework decides whether that call is permitted to run — and whether it gets logged.
Routing
Which model runs
In multi-model pipelines, the framework manages which model handles each step — sometimes using a lightweight LLM call to classify the task first. Perplexity and Copilot both do this. The framework controls the pipeline structure. Each model does the reasoning within its own step.
Logging
What gets recorded
The framework is the only place in the stack where the full sequence — assembled prompt, tool calls, model outputs, routing decisions — passes through in one place. If it doesn't emit that context to your security stack, it is gone after inference. No other layer can reconstruct it.
Your security stack — SIEM, DLP, IAM, audit — sits below all of this. It sees the access events that result from the framework's decisions. It does not see the decisions themselves unless the framework explicitly emits them.

Perplexity is a useful illustration. When you query Perplexity, their custom framework intercepts your request before any primary model call: a lightweight LLM may classify the query, the framework fetches live web results, assembles a grounded prompt with those results injected, selects which primary model is best suited, and on the way out enforces citation formatting. Microsoft Copilot does the same at enterprise scale, with its orchestrator grounding queries against Microsoft Graph (your emails, calendar, files, org structure) and routing to GPT, Claude, or Microsoft's own proprietary models depending on the task. The model is the brain at every step. What the framework controls is what each model sees, which model runs, and how outputs are shaped before they reach you. When that agent accesses sensitive data, your security team sees a sign-in log entry. They do not see the classification step, model selection, prompt assembly, or tool call sequence. The framework coordinated all of it. Your SIEM got the access event. The reasoning is gone.

Now scale that to your enterprise. You are not dealing with one framework. You are dealing with several — built by different vendors, using different architectural approaches, exposing different levels of telemetry. That variation is the security problem this article addresses.

A note on MCP

Model Context Protocol is the connective tissue beneath all four agent types — it's how frameworks expose tools to models. MCP shows up in F1, F2, and F3 alike, but its security properties are framework-dependent: a framework that uses MCP only inherits MCP's controls if the framework actually emits and enforces them. We address MCP as a cross-cutting concern in Part 3 of this series.

Four ways agents are built — and why it matters for security

The taxonomy that follows classifies agents by your relationship to the framework, not by who originally built it. From a security perspective, "framework you can instrument" is a fundamentally different posture than "framework someone else operates." Perplexity's engineers built their own custom framework on raw APIs — that's an F1-style build for them, but for your enterprise consuming Perplexity Enterprise, it's F4 because you cannot inspect or modify it. What matters is who can place controls, not who wrote the code.

Visual 2 — Four Ways You'll Encounter Agents in Your Estate
F1 Custom-Built Agent on Raw API
Examples: Internal tool calling Anthropic API directly · In-house chatbot on Azure OpenAI · Customer-facing LLM features built on raw API calls · Lambda functions invoking Claude or GPT directly

Your team calls the LLM API directly and builds the agent framework themselves: context assembly, tool definitions, routing logic, memory management, logging. There is no published framework providing structure. The application code is the framework. F1 spans a wide spectrum — from a thin wrapper around an API call to a serious custom orchestrator with full instrumentation. Both look identical in your network logs. The security posture is whatever your engineering team deliberately built. When teams ship F1 quickly to deliver a feature, governance is almost always opt-in — and almost always skipped.

F2 Agent Built on a Generic Framework
Examples: Agent built with Microsoft Agent Framework · Claude Agent SDK · Google ADK · smolagents · LangGraph · AutoGen · CrewAI

Your team uses a published framework — smolagents, LangGraph, Microsoft Agent Framework — to build their agent. The framework provides structure: memory management, tool invocation patterns, multi-agent coordination, workflow primitives. The LLM is still the reasoning brain at each step. The framework is what coordinates those steps. Crucially, these frameworks expose tracing hooks, run IDs, and step-level logging that can be wired to your security stack — but they are not on by default. You get the plumbing. You have to connect it.

F3 Proprietary Domain-Specific Agent
Examples: SAP Joule · Salesforce Agentforce · ServiceNow Now Assist · Workday Illuminate

Enterprise SaaS vendors have built their own proprietary frameworks on top of raw LLM APIs — not using Microsoft Agent Framework or smolagents, but their own purpose-built infrastructure. Joule's framework understands ERP workflows. Agentforce's framework understands CRM data models and Topics. The domain knowledge is baked into the framework design itself, not into the model. Governance is partially present but opaque — you cannot inspect the framework logic, and what telemetry the vendor exposes varies significantly by product and pricing tier.

F4 Platform-Managed Agent
Examples: Microsoft 365 Copilot · GitHub Copilot · Google Workspace Duet · Perplexity Enterprise

The framework is owned and operated entirely by the platform vendor. You are a consumer of their agent, not its builder. Model selection, context injection, tool routing, and action logging all happen inside infrastructure you don't control and cannot inspect. Your governance options are limited to what the vendor exposes through their admin console, compliance APIs, and audit log exports. The vendor's security posture becomes your security posture — for better or worse.

These four patterns coexist in almost every enterprise. F1 agents for internal tools, F2 agents built by engineering teams using frameworks, F3 agents embedded in enterprise SaaS, and F4 platform-managed agents used by all knowledge workers. Each is built differently. Each has different security properties. None of them are governed by the same controls.
The F4 Trap

Because F4 platforms manage their frameworks internally, security teams often treat them as "covered." They are not covered — they are partially covered by someone else's framework design, which may not align with your organization's risk posture, compliance requirements, or incident response procedures. The vendor's controls satisfy the vendor's defaults. They do not automatically satisfy yours. This is the single most common mistake in AI agent governance today.

The enterprise reality: you have all four agent types running right now

Here's what makes this hard in practice. The question is never "which type of agent are we dealing with?" It's always "which combination are we managing simultaneously?" Almost every enterprise above a certain scale has agents running across all four types at the same time — often without a complete inventory of them.

Six agent deployments. Four different agent types. Six completely different security postures — none visible through a single governance console. And this is conservative. Shadow agents built on F1 and F2 by individual teams are almost certainly present and uncounted.

The CISO looking at this estate has a fragmentation problem that no existing governance tool fully solves. Purview covers what Microsoft 365 Copilot's framework exposes. Salesforce Shield covers what Agentforce's framework exposes. SAP's audit log covers what Joule's framework exposes. Your SIEM covers what your custom agents emit. But none of them speak a common language, and none give you a unified view across all four types.

Your security posture for AI agents is only as strong as your weakest agent — and that's almost certainly one of the F1 or F2 agents you don't know about, deployed by a developer over a weekend.

What each agent type demands from your security team

Those six deployments produce essentially identical-looking access events in your SIEM — the difference is the depth of context behind each one, determined by which type produced it. Generic controls applied uniformly miss this entirely. They also miss something more fundamental: most agent security guidance today focuses on collecting logs and forwarding to a SIEM. Securing AI agents requires action at three distinct points in the lifecycle, and each agent type changes where you have leverage.

Visual 4 — The Three Security Pillars for AI Agents
Pillar 1
Secure by Design
Before the agent runs. Start with the architectural question: does this need to be agentic at all? Anthropic's guidance is direct — workflows (LLMs orchestrated through predefined code paths) are different from agents (LLMs dynamically directing their own process), and you should prefer workflows wherever the path is predictable. Use DAG and graph patterns (LangGraph's StateGraph, Microsoft Agent Framework's Workflow) to bound where agentic decision-making is allowed; use classifier-then-handler routing to keep the LLM in the lane-picking role while deterministic code does the driving. Every step that's deterministic code instead of LLM-decided is a step prompt injection cannot redirect. Then apply scope minimization on what remains: minimum tool surface, minimum data scope, per-agent identities with narrowly scoped credentials, prompt-injection-resistant system prompt design, vetted MCP servers and tool definitions, pinned framework and model versions, HITL checkpoints for irreversible actions, red-teaming before deployment.
Pillar 2
Runtime Security
While the agent is executing. Enforce policy at the moment of decision: intercept tool calls and allow/deny against a policy engine before they execute. Detect prompt injection in incoming context. Filter outputs for sensitive data and unsafe content. Apply rate limits and resource ceilings. Sandbox code-execution tools. Trip circuit breakers on anomalous behavior. Pause for approval on high-impact actions. This is the layer that prevents a bad decision from becoming a bad outcome.
Pillar 3
Security Operations
After the agent acts. Capture the full reasoning trace — assembled prompt, model response, tool calls, results — with session IDs that correlate multi-step sequences. Forward to your SIEM. Run output classification through DLP. Build detections for agentic-specific patterns: credential abuse by an agent, autonomous access escalation, anomalous tool sequences. Establish incident response procedures that include "shut the agent down" as a discrete action.
A complete agent security posture requires action at all three pillars. Most enterprises today address only the third — and only partially. Skipping pillars 1 and 2 means the only thing standing between a compromised agent and the blast radius is whatever the framework happened to enforce by default, which for most frameworks is nothing.

Shadow Agent Risk

F1 and F2 are the fastest types to deploy without formal approval — and the matrix above shows why that matters. For these types, all three pillars are your responsibility. When a developer ships a Lambda calling the Anthropic API, or a data scientist deploys a LangGraph agent over a weekend, none of the three pillars get built. Every uncounted F1 or F2 agent is one with nothing on any of the three pillars — no design review, no runtime guardrails, no telemetry. Inventory completeness is the prerequisite for security work to even begin.

F1 — Custom-Built Agent: You own the whole stack, no excuses

F1 Baseline — All Three Pillars

Secure by design: Start with the architecture. Decide which parts of the workflow actually need agentic decision-making and which can be deterministic code or a fixed graph — Anthropic's guidance is to prefer workflows over agents wherever the path is predictable, and that guidance applies here. Use a DAG or state graph to bound where the LLM is allowed to make decisions; let deterministic code drive the rest. Threat model the resulting agentic system with the same rigor as a public API. Define the smallest possible tool surface; deny-by-default for new tools. Use per-agent OAuth identities with narrowly scoped permissions. Design the system prompt to resist prompt injection (delimiters, role separation, output schema constraints). Define which actions require human approval before execution. Red-team prompt injection and tool misuse before going to production.

Runtime security: Intercept every tool call before execution and evaluate against a policy. Sandbox code-execution tools (Wasm, gVisor, isolated containers). Filter model outputs through DLP before they reach the user. Apply rate limits per session and per tool. Implement a kill switch that any on-call engineer can trip in under a minute.

Security operations: Log the full prompt, every tool call (name, parameters, response), and the model's final output, with a session ID tying them together. Forward to your SIEM with agentic-specific detection rules. The reasoning exists only at inference time — if you don't capture it then, it is gone.

F2 — Framework-Built Agent: Configure, enforce, and instrument what the framework exposes

F2 Baseline — All Three Pillars

Secure by design: Use the framework's workflow primitives, not just its agent primitives. LangGraph's StateGraph, Microsoft Agent Framework's Workflow, and OpenAI Agents SDK's handoff functions all let you express most of the control flow as a deterministic graph with agentic decisions only at specific nodes — that's a much smaller attack surface than a free-roaming agent loop. Use the framework's tool registration with explicit scopes — never auto-register all available tools. Configure per-agent identity (most frameworks support this; most teams use shared service principals anyway). Pin framework versions and model versions. Vet every MCP server and external tool the agent can call. Use the framework's HITL primitives (LangGraph's interrupt-and-resume, OpenAI Agents SDK's handoff approvals) for sensitive actions.

Runtime security: This is where F2's leverage is highest. Microsoft's Agent Governance Toolkit (open-source, MIT, released April 2026) plugs into Microsoft Agent Framework, LangChain, LangGraph, CrewAI, Google ADK, and OpenAI Agents SDK via their native extension points. Its Agent OS component is a sub-millisecond policy engine that intercepts every agent action before execution. Agent Mesh provides DID + Ed25519 cryptographic agent identity with the Inter-Agent Trust Protocol. Agent Runtime provides ring-isolated execution. Agent SRE adds circuit breakers and SLO enforcement. The toolkit maps to all ten OWASP Agentic Top 10 risks. If your organization is building agents on F2 frameworks, this is one of the fastest paths to structured runtime enforcement.

Security operations: Configure the framework's trace sink to forward to your SIEM. Microsoft Agent Framework emits per-step trace events. Claude Agent SDK exposes MCP connections, tool allowlists, and a hooks system. smolagents exposes full reasoning traces. LangGraph checkpoints by node. The hooks exist; most deployments leave them disconnected. The first question for any F2 agent: what trace sink is configured, and where are run IDs being collected?

F3 — Domain-Specific Agent: Configure aggressively, enforce contractually, log what's exported

You cannot change SAP Joule's runtime guardrails or Salesforce Agentforce's framework logic. What you can shape is configuration, contracts, and what you do with the exports.

Secure by design: Treat vendor admin configuration as a security control, not an enablement task. Disable Topics, skills, or connectors you don't need. Apply data-scope restrictions (Salesforce permission sets, SAP authorization objects) so the agent inherits only the access required. Require approval workflows for sensitive actions where the vendor exposes them. Document which guardrails the vendor enforces in their framework and which you've enabled — that's your residual risk register.

Runtime security: You have almost no direct lever here. Your contractual leverage is the substitute. Push vendors on: what runtime policy controls exist (allowlists, rate limits, kill switches), what triggers a circuit breaker, who can stop a misbehaving agent, and how fast.

Security operations: Audit-export questions to ask every F3 vendor:

VENDOR AUDIT ASSESSMENT — F3 DOMAIN-SPECIFIC AGENTS Decision Traceability → Does your audit log record the reasoning context, or only the outcome? → Can we retrieve the model's tool call sequence for a given agent session? → What is the retention period for agent reasoning traces? Intent Classification → Does your log distinguish user-directed from model-initiated data access? → When an agent autonomously accesses additional resources, is that flagged separately? Action Chain Correlation → Is there a session or trace ID that groups multi-step agent actions? → Can that ID be exported to a SIEM in real time or near real time? → What tier is required for full audit log access?

If a vendor can't answer these questions, treat that as a finding. The absence of agent-level runtime controls and telemetry is a gap in your ability to prevent and investigate incidents involving that agent.

F4 — Platform-Managed Agent: Configure to the maximum, accept the rest as residual risk

Secure by design: Sensitivity labels, indexed data scope, conditional access policies, agent-level permissions in admin tooling — these are your only design-time levers. Configure them to the maximum the platform allows. For Microsoft 365 Copilot specifically, sensitivity labels and Purview DLP policies inherited by Copilot are the closest you get to design-time control.

Runtime security: You have none directly. The vendor enforces (or doesn't enforce) policy in their infrastructure. Your only runtime lever is disabling the agent or revoking access — a binary, not a control.

Security operations: Maximize what the vendor exposes. Configure Purview audit forwarding to your SIEM. Enable GitHub Copilot audit logs. Establish a documented baseline of "sufficient audit evidence" for each platform before an incident forces the question. Document vendor coverage gaps as residual risk in your AI system registry — don't let "the vendor handles it" stand in for a control mapping.

The Punchline

There isn't one agent framework. There are four distinct types running in your enterprise right now, and one-size-fits-all controls fit none of them. The shape of your security work — what you can enforce at design time, intercept at runtime, and capture afterwards — depends entirely on which type you're dealing with.

Whichever you're dealing with, the question to ask first is the same: do we have controls at all three pillars, or only at the third? Most enterprises answer "only at the third, and only partially." Closing that gap — type by type, pillar by pillar — is the work this series exists to enable.

Start with your inventory. You cannot govern what you haven't counted.

What this means for you

Security Engineer

Build an agent inventory before anything else, classified by type (F1–F4). For each F1 and F2 agent, audit all three pillars: secure by design (tool scope, identity scope, system prompt review), runtime guardrails (is anything intercepting tool calls before execution?), and SecOps (are prompts, tool calls, and session IDs being logged?). Pillar 2 — runtime guardrails — is where most teams have nothing in place at all.

Security Architect

Design type-specific security requirements across all three pillars, not a single uniform standard. For F1/F2, secure-by-design starts with an architecture choice: where does the system actually need agentic decision-making, and where can deterministic graphs or workflows do the work? That choice shrinks the attack surface more than any runtime guardrail. From there: tool allowlists, scoped identities, HITL checkpoints; runtime enforcement requirements (policy engine, sandboxing, kill switch); observability specs. For F3/F4: configuration baselines for vendor admin tooling, contractual requirements for runtime controls, and minimum audit export thresholds. One policy applied uniformly across all four types and all three pillars is unenforceable.

AppSec Engineer

For F1 and F2 agents, treat tool definitions as a primary attack surface — equivalent to public API endpoints. Each tool binding is a capability the model can invoke autonomously. At design time: validate input/output schemas, enforce per-tool authorization, deny-by-default. At runtime: intercept calls, sandbox code execution, filter outputs through DLP. Red-team prompt injection and tool misuse before production. For F3 and F4, assess whether the vendor's tool governance prevents overprivileged model access to backend systems.

GRC Professional

Add agent type as a classification dimension in your AI system registry, then map controls across all three pillars per type. For F1/F2, your control evidence is internal: secure-by-design artifacts (threat models, tool scoping decisions), runtime configuration (policy engine deployment), audit logs. For F3/F4, your evidence is vendor-supplied: configuration attestations, vendor SOC reports covering their runtime controls, audit log exports. Where the vendor doesn't provide evidence for a pillar, document it as residual risk — don't let "the vendor handles it" substitute for a control mapping.

Auditor

Your evidence scope depends on agent type and pillar. For F1/F2: request threat models and tool-scoping decisions (design), policy engine configuration and runtime enforcement evidence (runtime), and prompt + tool call + session ID logs (SecOps). For F3/F4: request vendor configuration baselines, vendor attestations of runtime controls, and audit log exports at the contracted tier. The "right" evidence is different at each pillar; asking only for logs misses two-thirds of the control surface.

CISO

The hardest question in the next 12 months is not "are we using AI agents?" It's "for each agent in our estate, do we have controls at all three security pillars — design, runtime, and operations — or only at the third?" Shadow F1 and F2 agents typically have nothing on any pillar. Inventory first; then assess each agent against all three pillars; then close gaps in priority order — runtime guardrails are usually the highest-leverage missing piece.
